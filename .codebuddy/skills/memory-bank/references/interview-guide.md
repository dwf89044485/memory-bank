# 访谈指南

<!-- Memory-Bank by JosephDeng | interview-guide -->

当项目资料不足、需要通过对话收集信息时，按此指南进行。

## 1. 核心原则

1. **减少交互摩擦**：一次性提问，不要一句一句来
2. **编号清晰**：使用编号让用户可以按序回答
3. **允许跳过**：可选信息明确标注可跳过
4. **不像问卷**：保持自然对话感

## 2. 必问问题

**严格按照以下模板回复，不要添加额外问题或修饰语：**

```
需要以下核心信息：

1. 产品名称是什么？
2. 对标竞品有哪些？（或一句话描述产品价值）
```

### ⚠️ 重要约束

- **只问这 2 个必要问题**，不要问当前工作焦点、产品类型、目标用户、体验原则等可选信息
- **不要添加**"尽量一句话回答"等引导语
- **不要添加**示例回答或过多解释
- 保持简洁，总字数控制在 50 字以内

### 问题说明

| 问题 | 目的 | 示例回答 |
|------|------|----------|
| 产品名 | 如果是知名产品，AI 可以推导大量信息 | 飞书、Notion、自研 CRM |
| 对标竞品（或产品价值） | 即使产品不知名，竞品通常是市场领头产品，可推导用户群、场景；或通过一句话产品价值理解产品定位 | Notion、Confluence / 让团队知识协作更高效 |

## 3. 可选问题

**⚠️ 重要：可选问题不应该在初次收集时询问。**

可选信息（产品类型、目标用户、体验原则等）应该：
- 通过对标竞品推导
- 或者标注 `[待补充]`
- 用户可以在后续通过"更新记忆库"命令补充

**不要主动询问可选信息**，这会增加用户负担和交互摩擦。

## 4. 对话流程

### Step 1: 提问

严格使用第 2 节的回复模板，只问 2 个必要问题，不添加额外内容。

### Step 2: 接收回答

解析用户的批量回答，将回答与问题一一对应。

### Step 3: 确认理解

- 如果回答有歧义或不完整 → 针对性追问，不重复已回答的问题
- 如果回答清晰完整 → 直接进入推导阶段（读取 `references/reasoning-guide.md`），不询问可选信息

## 5. 回答解析

用户可能以多种格式回答：

### 格式 1：编号对应
```
用户：
1. 企业知识库
2. Notion、Confluence
```

### 格式 2：自然语言
```
用户：产品叫企业知识库，对标 Notion 和 Confluence，目前在优化搜索体验
```

### 格式 3：部分回答
```
用户：产品是企业知识库
```
→ 追问：对标竞品有哪些？或者一句话描述产品价值？

## 6. 特殊情况处理

### 6.1 用户不知道竞品
```
用户：没有明确的竞品

AI：没关系，可以告诉我这个产品大概是做什么的吗？
    比如：协作工具、数据分析平台、内容管理系统等
```

### 6.2 用户回答过于简略
```
用户：做个 CRM

AI：好的，CRM 系统。对标竞品有哪些？比如 Salesforce、HubSpot？
    或者一句话描述产品价值？
```

### 6.3 用户想跳过
```
用户：先跳过吧

AI：好的，将直接创建最小版本记忆库，缺失信息标注为"[待补充]"。
    后续可以随时补充。
```
