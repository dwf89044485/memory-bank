# 访谈指南

当项目资料不足、需要通过对话收集信息时，按此指南进行。

## 1. 核心原则

1. **减少交互摩擦**：一次性提问，不要一句一句来
2. **编号清晰**：使用编号让用户可以按序回答
3. **允许跳过**：可选信息明确标注可跳过
4. **不像问卷**：保持自然对话感

## 2. 必问问题

**严格按照以下模板回复，不要添加额外问题或修饰语：**

```
需要以下核心信息：

1. 产品名称是什么？
2. 对标竞品有哪些？
3. 当前工作重点是什么？
```

### ⚠️ 重要约束

- **只问这 3 个必要问题**，不要问产品类型、目标用户、体验原则等可选信息
- **不要添加**"尽量一句话回答"等引导语
- **不要添加**示例回答或过多解释
- 保持简洁，总字数控制在 50 字以内

### 问题说明

| 问题 | 目的 | 示例回答 |
|------|------|----------|
| 产品名 | 如果是知名产品，AI 可以推导大量信息 | 飞书、Notion、自研 CRM |
| 对标竞品 | 即使产品不知名，竞品通常是市场领头产品，可推导用户群、场景 | Notion、Confluence |
| 当前焦点 | 聚焦具体子场景，让推导更精准 | 优化多维表格体验、准备需求评审 |

## 3. 可选问题

**⚠️ 重要：可选问题不应该在初次收集时询问。**

可选信息（产品类型、目标用户、体验原则等）应该：
- 通过对标竞品推导
- 或者标注 `[待补充]`
- 用户可以在后续通过"更新记忆库"命令补充

**不要主动询问可选信息**，这会增加用户负担和交互摩擦。

## 4. 对话流程

```xml
<interview_flow>
  <step name="提问">
    <action>严格使用第 2 节的回复模板</action>
    <constraint>只问 3 个必要问题，不添加额外内容</constraint>
  </step>

  <step name="接收回答">
    <action>解析用户的批量回答</action>
    <action>将回答与问题一一对应</action>
  </step>

  <step name="确认理解">
    <if condition="回答有歧义或不完整">
      <action>针对性追问，不重复已回答的问题</action>
    </if>
    <if condition="回答清晰完整">
      <action>直接进入推导阶段（读取 reasoning-guide.md）</action>
      <action>不询问可选信息</action>
    </if>
  </step>
</interview_flow>
```

## 5. 回答解析

用户可能以多种格式回答：

### 格式 1：编号对应
```
用户：
1. 企业知识库
2. Notion、Confluence
3. 优化搜索体验
```

### 格式 2：自然语言
```
用户：产品叫企业知识库，对标 Notion 和 Confluence，目前在优化搜索体验
```

### 格式 3：部分回答
```
用户：产品是企业知识库，竞品是 Notion
```
→ 追问：当前工作焦点是什么？

## 6. 特殊情况处理

### 6.1 用户不知道竞品
```
用户：没有明确的竞品

AI：没关系，可以告诉我这个产品大概是做什么的吗？
    比如：协作工具、数据分析平台、内容管理系统等
```

### 6.2 用户回答过于简略
```
用户：做个 CRM

AI：好的，CRM 系统。对标竞品有哪些？比如 Salesforce、HubSpot？
    当前工作焦点是什么？
```

### 6.3 用户想跳过
```
用户：先跳过吧

AI：好的，将直接创建最小版本记忆库，缺失信息标注为"[待补充]"。
    后续可以随时补充。
```
